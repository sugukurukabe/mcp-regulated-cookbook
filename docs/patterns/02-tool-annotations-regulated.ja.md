# パターン 02: 規制業務向けツール・アノテーション

> **一文要約.** `destructive` (破壊的)、`idempotent` (冪等)、`read-only` (読み取り専用)、`open-world` (オープンワールド) の各アノテーションは、エンジニアにとって可逆的に感じるかどうかではなく、「そのツールの法的な効力」に基づいて選択してください。

## いつこのパターンを使うべきか

以下の条件に当てはまる場合、このパターンの対象となります。

- MCP サーバー内のツールが誤って呼び出された際、規制当局の定義による報告対象のインシデント（提出期限の超過、不正開示、書類手続きなしでは取り消せない取引など）を引き起こす可能性がある。
- MCP クライアントに対しツールを公開しており、クライアント側がアノテーションのフラグを見てユーザーに確認を求めるかどうかの判断を行っている。

以下の場合は**対象外**です。

- ツールがすべて公開データの読み取りのみであり、書き込みパスが存在しない。
- 業務ドメインに「不可逆性」という概念が存在しない。

## 課題（Forces）

- **エンジニアの直感 vs 法的効力:** 「データベースに書き込むが、明日その行を DELETE すれば済む」というのはエンジニアにとっては可逆的に感じます。しかし規制当局にとって、それはすでに発生した記録保持イベントであり、なかったことにはできません。
- **クライアントの UX vs 安全性:** あまりに多くのツールを `destructive` とマークすると、ユーザーは確認ダイアログを反射的にクリックするようになります（オオカミ少年化）。

## 解決策

**推奨事項の概要.** `destructiveHint`（破壊的ヒント）は「技術的な操作」ではなく、「現実世界での結果・影響」のプロパティとして扱ってください。「元に戻す」ためにシステム外の操作（電話連絡、紙のフォーム、規制当局への通知など）が必要となる場合、そのツールは `destructive: true` となります。

### 判定プロセス

**1. ツールの唯一の副作用はデータの読み取りであり、記録システムへの書き込みは一切発生しないか？**
はいの場合: `readOnlyHint: true`。`destructiveHint: false`、`idempotentHint: true` に設定します。

**2. もしこのツールが1度だけ誤って呼び出された場合、それを元に戻すには、同じ MCP サーバー内の逆のツール呼び出し以外の「人間のアクション」が必要になるか？**
はいの場合: `destructiveHint: true`。

**3. このツールがまったく同じ入力で2回呼び出された場合、2回目の呼び出しは1回目と同じ観測可能な結果を生み出すか？**
はいの場合: `idempotentHint: true`。

**4. このツールは、自社の管理外のソース（外部API、外部サイト等）から取得したデータに作用するか？**
はいの場合: `openWorldHint: true`。これを `destructiveHint: true` と組み合わせると、ユーザーに確認を要求すべきという最も強いシグナルをクライアントに送ることになります。

### 規制領域における実践例

| ツール名（例） | `readOnly` | `destructive` | `idempotent` | `openWorld` | 理由 |
|---|---|---|---|---|---|
| `lookup_corporate_number` (法人番号検索) | ✓ | false | ✓ | ✓ | 公開レジストリの読み取り。副作用なし。 |
| `submit_application_to_immigration_bureau` (入管への申請提出) | false | **true** | false | ✓ | 提出された申請を取り下げるには書類手続きが必要。ツールの呼び出しだけでは元に戻せない。 |
| `record_periodic_report_filing` (定期届出の記録) | false | **true** | ✓ | false | 再呼び出しに対しては冪等だが、規制上の届出を記録するという行為そのものが報告イベントとなるため。 |

## このパターンが失敗するケース

- **失敗モード: 規制当局が「破壊的」の定義を変更する。** 新しい法律やガイドラインにより、無害に見えていたツールが報告対象イベントに変わる可能性があります。
- **失敗モード: クライアント UX がアノテーションを骨抜きにする。** 一部のクライアントには「このサーバーが何でも実行することを許可する」という包括的なトグルがあり、ツールごとの確認がスキップされます。

## 本番環境での採用実績

- [**Case Study 01: スグクル株式会社**](../case-studies/01-sugukuru.ja.md) — TypeScript の `suguvisa-mcp` サーバーはビザ自動化のために42のツールを維持管理しており、ツールのアノテーションを広範に活用しています。CI のスクリプト (`audit-tool-annotations.ts`) を使用して、日本国の出入国管理法が要求する基本的な規制意識のベースラインをアノテーションが満たしていることを保証しています。
